# SimpleCom-server-core 项目分析文档

## 1. 总体定位与技术栈

- **项目类型**：纯 Java 的「服务器核心库」，专门负责 SimpleCom 的服务端核心逻辑（端口复用代理、WebSocket 认证与语音/信道管理），供 Bukkit/Fabric 等平台插件或模组调用。
- **构建工具与依赖**：
  - 使用 **Gradle + Java 8**（`build.gradle` 中 `JavaVersion.VERSION_1_8`）。
  - 主依赖只有 **SnakeYAML**（配置文件解析），以及 JUnit5 测试依赖。
  - 通过 `maven-publish` 插件发布为 `simplecom-server-core` 库。

## 3. 配置系统（`com.xiaofan.config`）

### 3.1 SimpleComConfigLoader

- 使用 SnakeYAML 加载 `SimpleComConfig/config.yml`。
- 第一次启动如果没有配置文件，会自动生成一个带详细中英文注释的默认配置。
- 支持外部传入配置根目录：
  - 上层平台在启动时调用 `setConfigBaseDir(File baseDir)`，例如 Bukkit 传入「服务器根目录」，Fabric 传入「游戏目录」；否则会退回到 JAR 所在目录或 `user.dir`。
- 解析后的配置封装为 `ProxyConfig` 对象返回。

### 3.2 ProxyConfig（配置内容）

主要字段：

**代理相关**

- `proxyEnabled`：是否启用「端口复用代理」
  - `true`：开启一个「复用端口」，把 MC 和 WS 代理到同一个对外端口上。
  - `false`：不启用复用代理，但**内部 WebSocket 服务器仍然运行**，此时客户端要直接连 `ws-port`。
- `bindAddress` / `listenPort`：复用代理对外监听地址与端口。
- `mcBackendHost` / `mcBackendPort`：后台真正的 MC 服务器地址与端口。
- `wsPort`：内部 WebSocket 服务器端口（一般只绑定 127.0.0.1）。
- `requireProxyConnection` / `allowedSourceAddresses`：限制只有来自指定源地址的连接才被视为「可信代理」。

**音频传输性能开关**

- `compressionEncoder`：是否启用压缩编码器，减少带宽但可能导致音质异常。
- `lowLatency`：低延迟模式（实验性），追求实时性但可能有卡顿。

**Voice Stream API（对接其他语音服务，如 KOOK/Discord）**

- `voiceApiEnabled` / `voiceApiToken` / `voiceApiPort` / `voiceApiBind`。
- 如果 `enabled=true` 且 `token` 为空，直接抛异常阻止启动，强调安全性。

配置模块**完全与 Bukkit/Fabric 解耦**，只处理文件和 YAML，方便在多平台复用。

## 4. 网络结构总览

### 4.1 整体架构

1. **外部世界**：玩家客户端只看到**一个对外端口**（`listen-port`），走 TCP：
   - 如果是 HTTP/WebSocket 握手，则认为是**语音/Web 控制连接**。
   - 否则认为是**MC 协议连接**。

2. **PortMultiplexProxy**（`com.xiaofan.proxy.PortMultiplexProxy`）：
   - 监听 `bindAddress:listenPort`，根据首包判断走 MC 还是 WS。
   - MC 方向：转发到真实 MC 服务器，同时在握手包里「注入真实 IP」。
   - WebSocket 方向：转发到本机的 `SimpleComWsServer`（127.0.0.1:`wsPort`），并在前面注入一行 `X-Real-IP: xxx`。

3. **内部 WebSocket 服务器 SimpleComWsServer**：
   - 只绑定在 `127.0.0.1:wsPort`。
   - 手动实现 HTTP 升级握手和 WebSocket 帧协议。
   - 完成验证码认证、语音数据转发、信道/加密信道管理等逻辑。

4. **状态与业务模块（`com.xiaofan.ws` 包）**：
   - `WsConnectionRegistry`：管理已认证连接、用户信道、断线重连逻辑。
   - `VerificationCodeStore`：验证码生成与校验。
   - `SimpleComWsHandler`：验证码认证逻辑的纯业务函数。
   - `EncryptedChannelManager`：加密语音信道的创建/验证/清理。
   - `WsSender`：对单个 WS 连接的封装（发送文本/二进制）。
   - `SeedMessageApi` / `SeedMessageResult`：种子消息（对外 API）。

## 5. 端口复用代理（PortMultiplexProxy）

### 5.1 功能定位

「**一个对外端口上同时承载 MC 与 WebSocket**」，只开放一个公网端口，适合内网穿透/小服务器。

### 5.2 工作流程

1. **监听与 accept 循环**：用 `ServerSocket` 监听 `bindAddress:listenPort`，使用 `CachedThreadPool` 处理每个连接。
2. **首包判别协议**：读取前 4 个字节。如果是 `"GET "` → 认为是 HTTP/WS 请求，走 `forwardToWsServer`；否则走 `forwardToMinecraft`。
3. **WebSocket 转发（forwardToWsServer）**：
   - 读到完整 HTTP 请求头（停在空行）。
   - 建一个到 `127.0.0.1:wsPort` 的 socket。
   - 向后端先写一行：`X-Real-IP: <client-ip>\n`，再写原始请求头。
   - 把 WS 后端返回的 HTTP 握手响应首部完整地回写给客户端。
   - 把双方 socket 放入双向转发循环（一个方向异步，另一个方向同步）。
   - 这样**内部 WS 服务器就能拿到真实 IP**（通过前面的那一行）。
4. **MC 转发（forwardToMinecraft）**：
   - 与 MC 后端建立连接。
   - 读 MC 握手包，解析 VarInt 长度和字段，**替换其中的「地址」字段为真实客户端 IP**（MC 服务器看到的 IP 仍是玩家真实 IP）。
   - 额外再读一个 Login Start 包，解析出用户名，用于日志。
   - 然后建立双向流复制，直到一端关闭。

### 5.3 实现特点

- 自己实现了 MC 协议中的 VarInt 读写和简单包解析（只解析握手包和 login start 里的用户名）。
- 所有 I/O 都是基于 `Socket`/`InputStream`/`OutputStream`，不依赖 Netty/Bukkit 等框架。
- 代理自身可通过 `proxyEnabled` 开关完全禁用，此时只起 WebSocket 服务器。

## 6. 内部 WebSocket 服务器（SimpleComWsServer）

### 6.1 功能定位

「**专用 WebSocket 认证和语音中继服务器**」，只监听 `127.0.0.1:wsPort`，一般由 PortMultiplexProxy 反向接入。

### 6.2 握手流程

1. 先读第一行。若第一行以 `X-Real-IP:` 开头（从代理转过来），则取出真实 IP，再继续读真正的 HTTP 请求头（直到空行）；否则把第一行当作标准 HTTP 请求的第一行继续读。
2. 从请求头字符串中提取 `Sec-WebSocket-Key`，计算 `Sec-WebSocket-Accept`：`accept = Base64(SHA1(key + WS_MAGIC))`。
3. 回写完整的 Upgrade 响应报文（101 Switching Protocols + Upgrade: websocket + Connection: Upgrade + Sec-WebSocket-Accept）。
4. 设置 socket 无限超时，进入帧循环 `wsFrameLoop`。

### 6.3 帧循环（wsFrameLoop）核心逻辑

**内部状态**：`authenticated`、`authenticatedUser`、`pingTask`、`WsSender sender`。

**支持的 opcode**：

- **8（CLOSE）**：回复 Normal closure 并退出。
- **9（PING）**：回复 PONG。
- **1（TEXT 帧）**：
  - **未认证**：文本应为 `{"username":"xxx","code":"nnnnnn"}`，解析后调用 `SimpleComWsHandler.authenticate`：
    - OK：注册连接、发送认证成功消息、服务器状态、初始用户列表，并启动每 6 秒的 ping + 用户列表推送。
    - NOT_FOUND：close code 4004（验证码不存在或过期）。
    - CODE_MISMATCH：close code 4003（验证码错误）。
  - **已认证**：文本帧用于控制指令（信道切换、加密信道列表/连接/创建等）。
- **2（BINARY 帧）**：仅在已认证状态下当作「语音数据包」，解析包头后调用 `forwardVoicePacket`，只转发给**同信道的其他已认证用户**；信道 0 不转发。

**连接关闭时**：取消 ping 任务、从 `WsConnectionRegistry` 注销、广播该信道用户列表更新；若信道号 > 100（加密信道），调用 `EncryptedChannelManager.checkAndCleanup`。

### 6.4 信道与加密信道

- **普通信道**：默认新认证用户置于信道 1；文本消息 `{"type":"channel","channel":X}` 切换信道，返回 `channelok` 和当前信道 `userlist`，并广播旧/新信道的用户列表。
- **加密信道（信道号 > 100）**：由 `EncryptedChannelManager` 管理；支持请求列表、连接（验证名字+密码哈希）、创建；成功后切换用户信道并推送 `userlist`，离开时检查清理。

## 7. 认证与会话管理（com.xiaofan.ws）

- **SimpleComWsHandler**：验证码 + 用户名的认证逻辑，调用 `VerificationCodeStore.validateCode`，根据结果返回 NOT_FOUND / CODE_MISMATCH / OK；提供轻量 JSON 解析 `parseUsername` / `parseCode`。
- **VerificationCodeStore**：按用户名生成一次性验证码并存储，有过期时间；`validateCode` 校验并消费；断线重连时由 `WsConnectionRegistry.tryNotifyReconnect` 生成新验证码。
- **WsConnectionRegistry**：
  - 单例；维护 `usernameToSender`、`usernameToChannel`。
  - 平台集成：`PlayerOnlineChecker`（玩家是否在线）、`ReconnectNotifier`（通过游戏内聊天发送新验证码）。
  - 关键方法：`register`、`setChannel`/`getChannel`/`getUsersInChannel`、`getSender`/`isConnected`/`getAllSenders`、`closeForUser`（玩家退出 MC 时主动关 WS）、`unregister(username, playerQuitMc)`（WS 断线时若玩家仍在线则生成新验证码并通知重连）。
- **WsSender**：封装 socket + outputStream，提供 `sendText`/`sendBinary`/`isOpen`/`closeSocket()`。

## 8. 总体架构总结

本项目是一个「**轻量协议网关 + 语音/信道控制服务**」，与 MC 插件主体解耦：

- 使用**端口复用代理**在一个公网端口上同时承载 MC 和 WebSocket；
- 对 MC 协议只做最小侵入（透传 + 修正 IP + 打日志）；
- 对 WebSocket 完全自实现握手与帧解析，在其上实现：
  - 基于验证码的登录认证；
  - 语音数据在同信道玩家间的转发；
  - 信道/加密信道/在线用户列表的管理与广播；
  - Voice Stream API 的配置（对接外部语音平台）。

**与上层（Bukkit/Fabric/其它服务）的边界**：

- 通过 `SimpleComConfigLoader.setConfigBaseDir` 控制配置位置；
- 通过 `WsConnectionRegistry.setPlayerOnlineChecker` 和 `setReconnectNotifier` 把「玩家在线状态」和「如何提示验证码」交给平台；
- 其余逻辑保持「纯 Java，零 Bukkit/Fabric 依赖」，便于多平台复用。
